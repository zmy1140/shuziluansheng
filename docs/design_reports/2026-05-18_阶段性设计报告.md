## 3. 项目背景

曲形管道内壁涂层磨抛加工存在几何结构复杂、接触状态难以直接观测、加工质量受多源因素影响等特点。对于内壁涂层表面质量控制而言，粗糙度 Ra 是重要评价指标之一，但其形成过程受到接触力、振动、声发射、主轴状态、工具路径、工件几何和材料特性等因素共同影响。传统加工监测通常侧重单一信号或离线检测，难以直观表达加工过程中的空间位置、工具运动状态和多源信号变化之间的关系。

本项目拟采用数字孪生前端作为实验监测与后续模型接入的载体。当前阶段不追求一次性建立完整工业数字孪生系统，而是先构建可本地运行的网页端监测与 3D 可视化框架，为后续传感器数据回放、Abaqus 仿真结果映射、粗糙度预测模型输出展示和加工过程演示提供统一界面基础。

## 4. 设计目标

### 4.1 总体目标

建立一个可在本地离线运行的曲形管道内壁涂层磨抛加工数字孪生前端原型，实现 3D 工位展示、多源监测数据展示、粗糙度预测结果预留和仿真/实验数据接入基础。

### 4.2 阶段目标

1. 搭建本地网页端前端工程，确保可启动、可测试、可构建。
2. 实现 Three.js 3D 视窗，支持 GLB/GLTF 模型导入、Draco 解码和自动居中。
3. 建立工况总览、切削力、振动、声发射、主轴状态和粗糙度预测等监测页面。
4. 明确当前所有数据为模拟数据，避免将占位功能误解为真实设备接入。
5. 建立 Abaqus 最小求解与 ODB 后处理链路，为后续真实仿真结果接入前端做预研。
6. 已实现“3D 加工动画 + 局部颜色渲染”的最小演示，使前端从静态模型展示过渡到加工过程可视化。
7. 已打通“本地 Abaqus 简化算例 -> JSON/CSV -> 前端颜色映射”的原型链路，为后续真实仿真或实验数据接入预留格式。

## 6. 方案选择与设计依据

### 6.1 前端技术方案

当前选择 Vite + JavaScript + Three.js。其依据如下：

1. Vite 启动速度快，适合本地原型迭代。
2. JavaScript 技术栈较轻，便于快速搭建浏览器端交互界面。
3. Three.js 支持 GLB/GLTF、相机控制、材质、网格、轨迹和后续颜色映射，适合作为数字孪生前端的 3D 可视化基础。
4. Vitest + jsdom 可对页面结构和关键 DOM 节点进行基础测试，降低界面改动造成入口丢失的风险。

### 6.2 数据路线方案

当前采用“模拟数据展示 -> 本地 CSV/JSON 回放 -> 真实采集或算法服务接入”的递进路线。该路线避免在设备链路未定型时过早绑定通信协议，同时可以先建立界面、字段、数据结构和展示逻辑。

### 6.3 采集链路初步判断

三轴加速度和声发射信号可能具有较高频率和较强瞬态特征，不宜简单依赖普通 PLC 模拟量模块完成原始波形采集。YE6275D 采集器可支持较高采样率振动采集，适合继续作为振动原始波形采集设备候选。PLC 更适合主轴状态、低频过程量、开关状态和执行控制。具体硬件链路仍需结合设备资料、供应商说明和实验室实际配置确认。

### 6.4 Abaqus 接入方案

当前已验证 Abaqus 2025 命令行可用，并通过最小 `.inp` 模型求解、ODB 后处理脚本导出 JSON/CSV。该方案说明后续可将 Abaqus 后处理结果转为前端可读取的数据格式。但当前 smoke test 仅用于验证软件链路，不代表真实磨抛接触仿真模型已经建立。

### 6.5 第二阶段 3D 加工动画方案

本轮选择“厚板工件 + 圆柱磨头 + 螺旋向内收缩轨迹 + 局部颜色色块”的最小方案。选择依据如下：

1. 厚板工件几何简单，便于先验证动画、轨迹和颜色映射逻辑，避免在真实曲形管道几何尚未确认时过早绑定复杂模型。
2. 圆柱磨头可以直观表达末端工具与工件表面的相对运动，后续可替换为真实磨头或蛇形臂末端结构。
3. 螺旋向内收缩轨迹可以展示覆盖过程、方向感和局部加工区域，适合作为导师或师兄汇报中的第一版动态演示。
4. 局部颜色下一版目标已明确为“温度场渲染示例”，用于表达加工区域标量场的空间分布；在真实热分析或测温数据确认前，不作为真实温度场、真实 Ra、真实应力或磨抛质量结论。
5. 导入 GLB 后采用模型包围盒拟合加工平面，使演示轨迹可贴近任意导入模型附近；该方法是几何近似，不是真实曲面投影、碰撞检测或接触求解。

### 6.6 仿真结果颜色映射方案

本轮选择先建立 Abaqus 简化厚板算例，而不是直接建立真实磨抛接触模型。主要原因是当前真实材料、接触、载荷、运动边界和粗糙度评价关系均待确认，若直接写成真实仿真容易造成科研结论失真。第一版仿真映射原型只验证数据链路：

1. Abaqus 负责生成一个可读取的结构化标量场。
2. ODB 后处理脚本负责导出 JSON/CSV。
3. 前端读取示例 JSON，并把坐标和标量值映射到厚板网格色块。
4. 历史链路验证中优先采用 Mises 应力，因为其在简化弹性板模型中容易提取、便于验证颜色映射；该指标不等同于温度场、粗糙度风险或真实磨抛结果。后续正式演示口径应改为温度场示例。

### 6.7 局部打磨温度场演示方案

本轮将 3D 演示从泛化的“加工动画”进一步收敛为“局部打磨温度场演示”。工件采用 `100 mm x 100 mm x 8 mm` 平板或展开面作为第一版对象，磨头参考直径采用 `24 mm`，工具模型固定加载 `public/models/tool.glb`。该方案选择的依据如下：

1. 真实曲形管道几何、涂层厚度和蛇形臂整体运动尚待确认，直接建立曲面接触演示会引入过多不确定因素。
2. `100 mm x 100 mm` 平板与 `24 mm` 磨头便于直观表达路径、接触区域和温度带相对尺寸，也便于后续建立 Abaqus 简化热源模型。
3. 温度网格采用 `40 x 40`，约对应 `2.5 mm x 2.5 mm` 的前端显示分辨率。该网格用于前端颜色映射，不等同于 Abaqus 有限元网格。
4. 工具模型保持固定加载，不在网页端改变真实模型文件；前端仅做归一化显示和底部接触贴合。

### 6.8 数据接口方案

本轮采用“外部程序导出 CSV，项目脚本转换为 JSON，前端只读取 JSON”的数据接口方案。路径和温度场分别采用两类 CSV：

1. 路径 CSV：`t_s,x_mm,y_mm,z_mm,nx,ny,nz,feed_mm_s`。
2. 温度场 CSV：`x_mm,z_mm,temperature_c`。

该方案避免前端直接依赖 Abaqus ODB、实验设备原始文件或同门路径规划程序内部格式。后续 Abaqus 后处理、实验测温整理或同门路径规划输出只需对齐 CSV 表头，即可接入当前前端。真实仿真和实验流程本身仍需验证，因此当前 CSV/JSON 只能作为接口格式和演示数据格式，不能直接解释为真实加工温度场。

| 边界条件 | 左侧固定，右侧给定位移 `0.02` | 简化拉伸，不是真实磨抛载荷 |
| 输出标量 | Mises 应力 | 历史链路验证用；下一版目标改为温度标量 |

前端坐标映射公式采用归一化网格索引：

```text
column = round(((x + width / 2) / width) * (columns - 1))
row = round(((z + depth / 2) / depth) * (rows - 1))
normalized = (value - minValue) / (maxValue - minValue || 1)
```

### 7.7 本轮局部打磨温度场关键参数

| 参数 | 当前值 | 说明 |
| --- | --- | --- |
| 工件/展开面尺寸 | `100 mm x 100 mm x 8 mm` | 第一版局部打磨演示基准，不代表真实曲形管道尺寸 |
| 磨头参考直径 | `24 mm` | 用于前端显示比例和后续简化热源仿真基准 |
| Three.js 映射比例 | `0.03` 场景单位/mm | 用于前端工件、路径和工具显示比例 |
| 温度网格 | `40 x 40` | 前端颜色映射网格，约 `2.5 mm x 2.5 mm`，不等同于有限元网格 |
| 固定工具模型 | `public/models/tool.glb` | 来源为用户提供的 `D:\Desktop\工具.glb`，当前文件大小为 14904 字节 |
| 路径 CSV | `data/demo/line_grinding_path.csv` | 字段为 `t_s,x_mm,y_mm,z_mm,nx,ny,nz,feed_mm_s` |
| 路径 JSON | `public/paths/line_grinding_path.json` | 由 `npm.cmd run convert:path` 生成 |
| 温度场 CSV | `data/demo/temperature_field.csv` | 字段为 `x_mm,z_mm,temperature_c` |
| 温度场 JSON | `public/simulation/temperature_field.json` | 由 `npm.cmd run convert:temperature` 生成 |
| 当前演示温度场 | `public/simulation/temperature_demo.json` | 演示数据，不代表真实 Abaqus 或实验测温结果 |

当前温度场 CSV 转 JSON 的核心结构为：

```json
{
  "valueKey": "temperature_c",
  "xKey": "x_mm",
  "zKey": "z_mm",
  "unit": "degC",
  "grid": {
    "columns": 40,
    "rows": 40,
    "widthMm": 100,
    "depthMm": 100
  },
  "samples": []
}
```

温度样本坐标映射到网格时，若字段为 `x_mm/z_mm`，前端优先使用 JSON 中的 `grid.widthMm/depthMm`，避免毫米坐标被误按 Three.js 场景单位映射。

### 8.9 问题排查与失败尝试

1. Abaqus 首次厚板输入出现 C3D8R 单元体积为零或负的问题。排查后判断为节点顺序与厚度方向不一致，已改为 Abaqus 中 X-Y 为板面、Z 为厚度，并在后处理中将 Abaqus Y 坐标映射为前端 z 坐标。
2. Abaqus Python 默认写 JSON 时存在中文编码不稳定风险。已在后处理脚本中使用显式 UTF-8 写入。
3. 浏览器自动化检查本地 Vite 页面时出现超时，本轮未形成自动截图证据；后续仍需进行人工或浏览器工具复核。
4. 固定工具模型加载后，场景中原有 cyan 小球容易被误认为运动工具。排查后确认它只是接触提示点，已删除。

6. 右侧“预测状态”卡片与顶部“粗糙度预测”指标卡重复。已删除右侧卡片，避免总览页重复表达同一个 Ra 数值。

## 11. 风险分析

1. 技术范围扩张风险：若同时推进前端、后端、PLC、采集卡、Abaqus 和预测模型，容易导致阶段目标失焦。
2. 数据真实性风险：当前模拟数据不能用于科研结论，后续报告必须区分“界面演示”和“真实实验结果”。
3. 采集链路风险：高频振动和声发射信号若采集设备选择不当，可能无法支撑 FFT、冲击特征或声发射事件分析。
4. 仿真可信度风险：Abaqus 最小测试只验证软件链路，真实磨抛接触模型需要材料、接触、边界条件和实验验证支撑。
5. 中文编码风险：终端显示乱码可能误导后续批量替换，需在浏览器或编辑器中确认。
6. 离线交付风险：当前更适合通过本地 Vite 服务运行，若要求任意电脑双击打开，需要单独验证静态包或桌面封装方案。
